# HiDock Knowledge App - Complete Redesign Specification

## Vision Statement

HiDock transforms ephemeral conversations into **permanent, searchable, actionable knowledge**. The app has two distinct modes:

1. **Device Management** - Basic utility for syncing audio from HiDock device (legacy purpose)
2. **Knowledge System** - The primary experience: capturing, organizing, exploring, and generating insights from knowledge

## Part 2: Assistant (formerly Chat)

### 2.1 Purpose

The Assistant is a **knowledge-powered AI** that:
- Answers questions using the knowledge base
- Generates insights by connecting disparate knowledge
- Produces artifacts (stored as Outputs)
- Maintains conversation history (itself becomes knowledge)

### 2.2 Key Features

1. **Conversation History** - All past conversations stored and searchable
2. **Context Injection** - Manually add knowledge items as context
3. **Artifact Generation** - Create outputs directly from conversation
4. **Save to Knowledge** - Store insights back to knowledge base
5. **Edit & Continue** - Branch conversations, edit previous messages

### 2.5 Assistant Actions

| Action | Description |
|--------|-------------|
| **Create Output** | Generate artifact from conversation (minutes, summary, etc.) |
| **Save Insight** | Store AI response as a knowledge item |
| **Add Context** | Inject knowledge items into conversation |
| **Edit Message** | Modify previous message and regenerate |
| **Branch** | Create new conversation from this point |
| **Delete** | Remove conversation from history |
| **Search History** | Find past conversations |

### 2.6 Conversation as Knowledge

Conversations are stored and become part of the knowledge base:

```typescript
interface Conversation {
  id: string
  title: string                    // AI-generated or user-set
  messages: Message[]
  context_ids: string[]            // KnowledgeCapture IDs used
  created_at: Date
  updated_at: Date

  // Knowledge integration
  generated_insights: string[]     // IDs of saved insights
  generated_outputs: string[]      // IDs of generated outputs
}

interface Message {
  id: string
  role: 'user' | 'assistant'
  content: string
  timestamp: Date

  // If user edited this message
  edited_at?: Date
  original_content?: string

  // If this spawned artifacts
  created_output_id?: string
  saved_as_insight_id?: string
}
```

## Part 3: Explore (formerly Search)

### 3.1 Purpose

Explore is the **discovery engine** - not just searching, but:
- Finding connections between knowledge
- Gathering context for the Assistant
- Identifying patterns across people, projects, events
- Preparing data for Output generation

### 3.2 Search Scope

| Scope | What's Searched |
|-------|-----------------|
| **Knowledge** | Transcripts, summaries, action items, decisions |
| **People** | Names, mentions, related knowledge |
| **Projects** | Project names, related knowledge |
| **Conversations** | Past Assistant conversations |
| **Outputs** | Generated artifacts |

## Part 4: People (formerly Contacts)

### 4.1 Purpose

People is the **person-centric view** of knowledge:
- Auto-populated from transcripts (speaker identification)
- Shows all knowledge related to a person
- Relationship mapping (who they interact with)
- Context for that individual

### 4.2 Auto-Population (Self-Populating)

People are **automatically created and enriched** - the user never manually creates them:

```typescript
// Auto-population triggers
const PEOPLE_AUTO_POPULATION = {
  // From calendar
  calendar_attendees: {
    trigger: 'Meeting synced with attendees',
    creates: 'Person with email, name from calendar',
    enriches: 'interaction_count++, last_seen'
  },

  // From transcription
  transcript_speakers: {
    trigger: 'AI identifies distinct speakers',
    creates: 'Person if new voice/name detected',
    enriches: 'knowledge_ids[], topics from transcript'
  },

  // From mentions
  transcript_mentions: {
    trigger: 'Name mentioned in transcript',
    creates: 'Person if new name detected',
    enriches: 'knowledge_ids[], context of mention'
  },

  // From action items
  action_item_assignee: {
    trigger: 'Action item assigned to someone',
    creates: 'Person if new name',
    enriches: 'pending_actions[], role inference'
  }
}
```

## Part 6: Actionables (formerly Outputs)

### 6.1 Purpose

Actionables is the **proactive action center** - not just generation, but **suggestions derived from knowledge**:
- Auto-populated with action items from meetings
- Suggests artifacts based on meeting type
- One-click generation of artifacts
- Tracks what needs to be done vs what's been done

### 6.2 Key Concepts

| Concept | Description |
|---------|-------------|
| **Actionable** | An intent to create something - the instruction/to-do |
| **Artifact** | The actual generated document (.md file) |
| **Suggestion** | AI-detected actionable from knowledge |

### 6.3 Auto-Population Logic

Actionables are **automatically populated** from:

```typescript
interface Actionable {

  // Result

  shared_at?: Date
```

```typescript
  project_id?: string
  person_ids: string[]

  type: 'team' | 'candidate' | 'customer' | 'external'

  company?: string

  topic_frequencies: Record<string, number>

  id: string

  status: 'active' | 'archived'
  description?: string

  person_ids: string[]
  created_at: Date
}

interface Conversation {
  id: string

  messages: Message[]
  context_ids: string[]
  created_at: Date
}

interface Output {
  id: string

  source_ids: string[]           // Knowledge IDs used

  device_present: boolean
  local_path?: string
  duration_seconds: number
}
```
