## Issue 1: Audio Playback Errors

### Problem
Console repeatedly showing:
```
[useAudioPlayback] Audio element error: {code: 4, message: 'MEDIA_ELEMENT_ERROR: Empty src attribute'}
```

### Root Cause
When stopping playback, we clear the audio element's `src` attribute by setting it to `''`. This triggers the audio element's error event (error code 4 = MEDIA_ERR_SRC_NOT_SUPPORTED) because the element now has no valid source. The error was being logged even though it's expected behavior.

### Solution
Modified the error event handler in `useAudioPlayback.ts` to ignore error code 4 when the src attribute is intentionally empty:

**File:** `src/hooks/useAudioPlayback.ts` (lines 95-111)

```typescript
audioRef.current.addEventListener('error', (e) => {
  const mediaError = audioRef.current?.error

  // Ignore error if src was intentionally cleared (code 4 = MEDIA_ERR_SRC_NOT_SUPPORTED with empty src)
  if (mediaError?.code === 4 && (!audioRef.current?.src || audioRef.current?.src === '')) {
    if (shouldLogQa()) console.log('[useAudioPlayback] Ignoring error from intentionally cleared src')
    return
  }

  // ... rest of error handling
})
```

### Impact
- Eliminates console spam from expected cleanup operations
- Real playback errors still logged and shown to user
- No functional change to playback behavior

## Issue 2: ResizablePanel Layout Warnings

### Problem
Console showing multiple warnings:
```
Invalid layout total size: 45%, 55%, 20%
Panel 'left-panel' has an invalid configuration: default size should not be greater than max size
```

### Root Cause
Panel sizes persisted from tablet mode (2-panel layout) were being used in desktop mode (3-panel layout), causing totals that exceed 100%. For example:
- Tablet mode: User resizes to [45%, 55%] for two visible panels
- Desktop mode: Tries to use [45%, 55%, 20%] = 120% total for three panels

Additionally, panel sizes could exceed min/max constraints after normalization.

### Solution
Added panel size normalization in `TriPaneLayout.tsx` for desktop mode:

**File:** `src/features/library/components/TriPaneLayout.tsx` (lines 33-59)

```typescript
// Normalize panel sizes to ensure they total 100% and respect constraints
const normalizeDesktopPanelSizes = (sizes: number[]): [number, number, number] => {
  const [left = 25, center = 45, right = 30] = sizes
  const total = left + center + right

  if (Math.abs(total - 100) < 0.1) {
    // Already normalized
    return [left, center, right]
  }

  // Normalize to 100% while respecting min/max constraints
  const scale = 100 / total
  const normalized = [
    Math.max(20, Math.min(35, left * scale)),   // left: min 20, max 35
    Math.max(30, center * scale),                // center: min 30, no max
    Math.max(20, Math.min(40, right * scale))    // right: min 20, max 40
  ]

  // Ensure final total is exactly 100
  const normalizedTotal = normalized[0] + normalized[1] + normalized[2]
  if (Math.abs(normalizedTotal - 100) > 0.1) {
    // Adjust center panel to make up the difference
    normalized[1] = 100 - normalized[0] - normalized[2]
  }

  return normalized as [number, number, number]
}

const desktopPanelSizes = normalizeDesktopPanelSizes(panelSizes)
```

Then use `desktopPanelSizes` instead of raw `panelSizes` for desktop layout `defaultSize` props.

### Impact
- Eliminates all ResizablePanel console warnings
- Smooth transitions between tablet and desktop layouts
- Panel sizes always respect min/max constraints
- No more invalid layout configurations

## Issue 3: Transcription Stuck in Progress

### Problem
Recordings showing "Transcription in progress..." indefinitely with no actual progress. Transcriptions never completing.

### Root Cause
**Two-part problem:**

1. **Silent processor failure**: The transcription processor (`transcription.ts:processQueue`) checks for Gemini API key on every run. If missing, it silently returns WITHOUT:
   - Marking pending/processing items as failed
   - Updating recording transcription_status
   - Notifying UI of the error

2. **No entry validation**: Users could queue transcriptions without an API key configured, leading to items that can never be processed.

### Solution

#### Part 1: Processor Error Handling

Modified `processQueue()` in `electron/main/services/transcription.ts` to properly handle missing API key:

**File:** `electron/main/services/transcription.ts` (lines 82-107)

```typescript
const config = getConfig()
if (!config.transcription.geminiApiKey) {
  console.error('[Transcription] Cannot process queue: Gemini API key not configured')

  // Mark all pending items as failed with clear error message
  const pendingItems = getQueueItems('pending')
  const processingItems = getQueueItems('processing')

  const allStuckItems = [...pendingItems, ...processingItems]
  if (allStuckItems.length > 0) {
    console.log(`[Transcription] Marking ${allStuckItems.length} stuck items as failed (no API key)`)

    for (const item of allStuckItems) {
      updateQueueItem(item.id, 'failed', 'Gemini API key not configured. Please add your API key in Settings.')
      updateRecordingTranscriptionStatus(item.recording_id, 'error')
      notifyRenderer('transcription:failed', {
        queueItemId: item.id,
        recordingId: item.recording_id,
        error: 'Gemini API key not configured. Please add your API key in Settings.'
      })
    }
  }

  return
}
```

**What this does:**
- Finds all stuck items (both 'pending' and 'processing')
- Marks each as 'failed' with clear error message
- Updates recording status to 'error'
- Notifies UI so user sees the failure
- Logs count of items marked failed

#### Part 2: Entry Point Validation

Added API key validation to both `queueTranscription` and `queueBulkTranscriptions` in `useOperations.ts`:

**File:** `src/hooks/useOperations.ts` (lines 31-41, 63-73)

```typescript
// Check if API key is configured before queuing
try {
  const apiKey = await window.electronAPI.config.getValue('transcription.geminiApiKey')
  if (!apiKey || apiKey.trim() === '') {
    toast({
      title: 'API key required',
      description: 'Please configure your Gemini API key in Settings before transcribing.',
      variant: 'error'
    })
    return false  // or return 0 for bulk
  }
} catch (e) {
  console.error('Failed to check API key:', e)
  toast({ title: 'Configuration error', description: 'Could not verify API key configuration', variant: 'error' })
  return false  // or return 0 for bulk
}
```

**What this does:**
- Checks API key BEFORE queuing any transcriptions
- Shows clear toast error if missing
- Prevents items from being added to queue when they can't be processed
- Applies to both single and bulk transcription operations

### Impact
- **Stuck transcriptions resolved**: All items stuck in "processing" state will be marked as failed on next processor run
- **Clear error messages**: Users see exactly why transcription failed and what to do (configure API key in Settings)
- **Prevention**: New transcriptions can't be queued without API key configured
- **User experience**: No more indefinite "in progress" states - failures are immediate and actionable
